<task type="auto">
  <name>Task 1: Migrate schema from v1 to v2</name>
  <files>src/db/schema.js, src/db/index.js</files>
  <action>
**schema.js changes:**
- Bump `SCHEMA_VERSION` from 1 to 2
- Update `SCHEMA_DDL` to include the full v2 schema (so fresh installs get v2 directly):
  - sessions table: add columns `file_size INTEGER`, `assistant_message_count INTEGER DEFAULT 0`, `fork_count INTEGER DEFAULT 0`, `real_fork_count INTEGER DEFAULT 0`, `is_compacted BOOLEAN DEFAULT 0`, `has_subagents BOOLEAN DEFAULT 0`, `last_updated_at TEXT`, `custom_title TEXT`, `slug TEXT`
  - messages table: add columns `parent_uuid TEXT`, `subtype TEXT`, `is_meta BOOLEAN DEFAULT 0`, `is_fork_branch BOOLEAN DEFAULT 0`
  - Add new tickets table:
    ```sql
    CREATE TABLE IF NOT EXISTS tickets (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      session_id TEXT NOT NULL,
      ticket_key TEXT NOT NULL,
      source TEXT NOT NULL,
      detected_at TEXT,
      is_primary BOOLEAN DEFAULT 0,
      FOREIGN KEY (session_id) REFERENCES sessions(session_id),
      UNIQUE(session_id, ticket_key, source)
    );
    CREATE INDEX IF NOT EXISTS idx_tickets_session ON tickets(session_id);
    CREATE INDEX IF NOT EXISTS idx_tickets_key ON tickets(ticket_key);
    ```
  - Add new import_log table:
    ```sql
    CREATE TABLE IF NOT EXISTS import_log (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      session_id TEXT,
      file_path TEXT NOT NULL,
      file_size INTEGER NOT NULL,
      imported_at TEXT NOT NULL DEFAULT (datetime('now')),
      status TEXT NOT NULL,
      error_msg TEXT,
      UNIQUE(file_path)
    );
    ```
- Export a new `MIGRATION_V1_TO_V2` string constant containing only the ALTER TABLE statements and CREATE TABLE statements needed to migrate an existing v1 database to v2

**index.js changes:**
- After the PRAGMA user_version check, add migration logic:
  - If `existingVersion === 1`: run `migrateV1toV2(db)` then set `PRAGMA user_version = 2`
  - The `migrateV1toV2` function wraps each ALTER TABLE ADD COLUMN in a try/catch (SQLite has no `ALTER TABLE ADD COLUMN IF NOT EXISTS`) so re-running is safe
  - Keep the existing behavior: if version is 0 (fresh) or matches SCHEMA_VERSION, apply DDL normally. If version is unknown and not 1 or 2, drop and recreate.
- Do NOT change the existing drop-and-recreate behavior for unknown versions

**Critical:** `node:sqlite` has NO `db.transaction()` method. Use `db.exec('BEGIN')` / `db.exec('COMMIT')` for the migration transaction.

<task type="auto">
  <name>Task 2: Create database writer module for import pipeline</name>
  <files>src/importer/db-writer.js</files>
  <action>
Create `src/importer/db-writer.js` exporting these functions that accept a `db` (DatabaseSync) parameter:

**`upsertSession(db, sessionData)`** — INSERT OR REPLACE into sessions table
- sessionData is an object with: session_id, project_id, file_path, file_size, file_modified_at, working_branch, primary_ticket, summary, custom_title, slug, first_message_at, last_message_at, message_count, user_message_count, assistant_message_count, tool_use_count, fork_count, real_fork_count, is_compacted, has_subagents
- Uses a prepared statement with `INSERT OR REPLACE` (sessions.session_id is UNIQUE)

**`insertMessages(db, sessionId, messages)`** — Batch insert messages in a transaction
- Wraps in `db.exec('BEGIN')` / `db.exec('COMMIT')` with try/catch for `db.exec('ROLLBACK')`
- Uses `INSERT OR IGNORE` (messages have UNIQUE(session_id, uuid))
- Each message object has: uuid, type, subtype, timestamp, parent_uuid, git_branch, is_meta, is_sidechain, is_fork_branch
- Uses a single prepared statement, loops over messages array

**`upsertTickets(db, sessionId, tickets, primaryTicket)`** — Insert ticket detections
- tickets is an array of {ticket_key, source, detected_at}
- Sets is_primary=1 for the ticket matching primaryTicket
- Uses `INSERT OR IGNORE` with UNIQUE(session_id, ticket_key, source)

**`updateImportLog(db, filePath, sessionId, fileSize, status, errorMsg)`** — Track import status
- Uses `INSERT OR REPLACE` on UNIQUE(file_path)
- Records session_id, file_size, status ('ok' or 'error'), error_msg

**`getImportedFileSizes(db)`** — Returns a Map of file_path -> file_size for all successfully imported files
- Query: `SELECT file_path, file_size FROM import_log WHERE status = 'ok'`
- Returns: `new Map()` of the results

**Critical patterns:**
- All batch operations (insertMessages) use manual BEGIN/COMMIT — `db.transaction()` does NOT exist in node:sqlite
- All prepared statements use `db.prepare()` with parameter binding (never string interpolation)
- upsertSession should be called OUTSIDE the messages transaction (single row, no batch needed)
  </action>
